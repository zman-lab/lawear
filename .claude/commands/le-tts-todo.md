# le-tts-todo — TTS 변환 진행 현황 관리

> Case 단위로 TTS 변환 진행 상태를 추적.
> Lv.1/2/3은 묶어서 1건으로 관리.
> 아규먼트 없음 — 호출하면 현황 표 출력.

## 호출 시 동작

1. `docs/tts_progress.md` 읽기 (없으면 초기 스캔 실행)
2. **자동 감지**: ttsData.ts 스캔 → 이미 변환된 데이터 있으면 "완료"로 자동 갱신
3. 과목별 요약 + 전체 진행률 출력
4. 다음 작업 자동 제안 (우선순위 기반)

### 출력 형식

```
## TTS 변환 진행 현황

| 과목 | 전체 | 완료 | 진행중 | 미착수 | 진행률 |
|------|------|------|--------|--------|--------|
| 민소 | 12   | 0    | 0      | 12     | 0%     |
| 민법 | 8    | 0    | 0      | 8      | 0%     |
| ...  |      |      |        |        |        |

### 다음 작업
> 민소 예비 미케01 Case01 (미착수)
```

## 데이터 파일: `docs/tts_progress.md`

### 구조

```markdown
# TTS 변환 진행 현황

> 마지막 스캔: {날짜}
> 전체: {N}건 | 완료: {N} | 진행중: {N} | 미착수: {N}

## 민사소송법

| ID | PDF 파일 | Case | 상태 | 게시판 | 비고 |
|----|----------|------|------|--------|------|
| 2026_minso_yebi_mike01_01 | 민소_예비_미케_01.pdf | Case01 | 미착수 | | |
| 2026_minso_yebi_mike01_02 | 민소_예비_미케_01.pdf | Case02 | 미착수 | | |

## 민법
...
```

### 상태값

| 상태 | 의미 |
|------|------|
| 미착수 | PDF 미변환 |
| 진행중 | Lv.1/2/3 변환 작업 중 |
| 리뷰대기 | 게시판에 올려서 사용자 확인 대기 |
| 완료 | 사용자 OK + 커밋됨 |

### ID 체계

`project_id_system.md` 준수:
- `{yyyy}_{과목영문}_{단계영문}_{문번}_{부수문번}`
- 예: `2026_minso_yebi_mike01_01`

### 상태 갱신 — 이중 트래킹

**A. le-dev가 작업 중 직접 갱신 (실시간)**
- **미착수 → 진행중**: TTS 변환 시작 시
- **진행중 → 리뷰대기**: 게시판에 Lv.1/2/3 + diff 올렸을 때
- **리뷰대기 → 완료**: 사용자 OK 후 커밋 시
- 게시판 글 ID도 함께 기록

**B. le-tts-todo 호출 시 자동 감지 (보정)**
- `ttsData.ts` 스캔 → 해당 Case 데이터가 있으면 "완료"로 자동 보정
- progress.md와 ttsData.ts 불일치 발견 시 → ttsData.ts 기준으로 갱신
- A에서 갱신 누락된 것을 B가 잡아줌

## 리스캔

사용자가 대화에서 "리스캔해줘", "PDF 추가했어" 등 요청 시:

```
1. ~/Downloads/2026_USB/2026_박문각_피뎁/ 재스캔
2. 기존 docs/tts_progress.md와 diff
3. 새로 발견된 PDF → "미착수"로 추가
4. 삭제된 PDF → 사용자에게 보고 (자동 삭제 안 함)
5. 기존 상태는 유지
6. ttsData.ts 자동 감지로 완료 상태 보정
```

## 우선순위 (다음 작업 제안)

CLAUDE.md의 과목별 우선순위 준수:

```
1순위: 논술 5과목
  민소 예비 → 민법 예비 → 형법 예비 → 형소 예비 → 부등법 입문
2순위: 민사서류 → 보류
3순위: 부등서류 → 보류
```

같은 과목 내에서는:
- 미케(미니케이스) > 모고(모의고사) > 순환
- 번호순 (01 → 02 → ...)
