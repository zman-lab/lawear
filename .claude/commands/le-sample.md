# le-sample — TTS 샘플 생성

> 과목/파일을 지정하여 TTS 텍스트를 변환하는 스킬.
> 최우선 규칙: AI 자의적 해석 금지 (R-09) — 원본 PDF 정보만 사용.
> 게시판 #462 피드백이 tts_rules.md보다 우선.

## 입력 예시

```
/le-sample 민소 미케01 Case1
/le-sample 형법 01 문제3
/le-sample 민법 02
/le-sample 부등법 미케03 Case1~3
```

## 최우선 규칙

```
R-09: AI 자의적 해석 금지
- 원본 PDF에 있는 정보만 사용
- 내용 추가/변경/보충/해석 절대 금지
- 기계적 변환(기호→텍스트, 한자→한글)만 허용
- 법률적 판단/해석 변경 절대 금지
- 애매하면 원본 표현 유지
- 이 규칙은 다른 모든 규칙보다 우선
```

## 변환 워크플로우

```
1. 입력 파싱
   - 과목/파일/Case 번호 분리
   - 부등서류 → "보류 과목" 안내 후 중단

2. PDF 경로 확인
   - ~/Downloads/2026_USB/{과목}/{파일}.pdf 존재 여부 확인
   - 없으면 목록 조회 후 사용자에게 선택 안내

3. PDF 텍스트 추출 (Phase 1)
   - PyMuPDF(fitz)로 텍스트 레이어 추출
   - python3 -c "import fitz; doc=fitz.open('{파일경로}'); print(chr(10).join([p.get_text() for p in doc]))"
   - 파싱 품질 낮은 과목(민사서류): 추출 결과 먼저 확인 후 진행

4. AI 전처리 (Phase 2) — tts_rules.md 기본 10개 + 추가 R-01~R-11 순서 적용
   Step 4-1. R-09 마킹 (자의적 해석 위험 구간 선별)
   Step 4-2. 기계적 정제 (R-01~R-05, R-08)
     - 판례번호 완전 제거 (R-01)
     - 날짜 정규화 + 상대적 시간 표현 (R-02)
     - 괄호 → "혹은" 변환 (R-03)
     - 콜론 → 쉼표 변환 (R-05)
     - 특수기호 제거 (R-08): [] — : () /
     - 꺽쇠 인용부호 제거 (기본 규칙 3)
     - 특수 띄어쓰기 정리 (기본 규칙 6)
     - 한자 병기 제거 + 인명 甲乙丙 → 갑/을/병 변환 (기본 규칙 1, 7)
     - 페이지 헤더/푸터 제거 (기본 규칙 4)
     - 빈칸/○○○ 처리 (기본 규칙 10)
     - 원문자 번호 통일: ① → 첫째 또는 1번 (기본 규칙 8)
   Step 4-3. 과목별 특화 전처리 적용

5. 과목별 특화 규칙 적용 (Phase 3)
   - 민소/민법: 빈칸 박스 처리 전략 확인 (사용자에게 선택 요청 또는 기존 방식 유지)
   - 형법/형소: 별표(★★) → "중요" 변환, 중첩 괄호 단순화
   - 부등법: "명문x" → "명문 규정 없음" 변환
   - 민사서류: 등기부/가족관계 표 → AI 줄글 재구성

6. TTS 텍스트 생성 (Phase 4) — sample_format.md 출력 형식 적용
   - 문제: 키워드 포함 요약 (답안 구성 키워드 반드시 포함, 날짜 상대적 표현)
   - 목차: 원문 그대로 (요약/재구성 금지 — R-07)
   - 답안: 목차 순서에 맞게 구성, 호흡 조절 쉼표 삽입 (R-04)
   - 문제 → 목차 → 답안 순서 (R-06)

7. diff 생성 (R-10)
   형식:
   원본: {원문}
   TTS:  {변환 결과}
   (변경 없는 부분은 생략)

8. QA — 듣는 사람 입장 검증 5개 항목 (R-11)
   1. 이 부분이 구분되어 들리는가?
   2. 문장이 너무 길어서 맥락 이해나 암기가 힘들지 않은가?
   3. 날짜/인명이 혼동되지 않는가?
   4. 법률 용어가 정확히 전달되는가?
   5. AI가 자의적으로 해석/보충한 부분이 없는가?
   → 문제 발견 시 수정 후 재출력

9. 체크리스트 10개 증빙 제출 (le-review로 전달용)
   1. 판례번호 제거 완료 — 처리 건수 또는 "해당 없음"
   2. 날짜 정규화 완료 — 처리 건수 + 상대적 표현 적용 여부
   3. 꺽쇠 인용부호 제거 완료 — 처리 건수 또는 "해당 없음"
   4. 특수 띄어쓰기 정리 완료 — 처리 건수 또는 "해당 없음"
   5. 한자 병기 제거 완료 — 처리 건수, 인명 변환 여부
   6. 페이지 헤더/푸터 제거 완료 — 처리 건수 또는 "해당 없음"
   7. 빈칸/○○○ 처리 완료 — 처리 방식 명시
   8. 과목별 특화 전처리 적용 완료 — 적용 규칙 명시
   9. 원본 대비 정보 누락 없음 확인 — 확인 방법 명시
   10. TTS로 읽었을 때 자연스러운 문장 확인 — QA 결과 요약
```

## 출력 형식 (sample_format.md 기준)

```
문제,
{키워드 포함 요약 — 답안 구성 키워드 반드시 포함, 날짜 상대적 표현}

목차,
{원문 그대로}

답안,
{목차 순서에 맞게 구성, 호흡 조절 쉼표 포함}

---

diff (원본 vs TTS),
원본: {원문}
TTS:  {변환 결과}
...

---

QA 결과,
1. 구분: {판정}
2. 문장 길이: {판정}
3. 날짜/인명: {판정}
4. 법률 용어: {판정}
5. R-09 준수: {판정}

---

체크리스트 증빙,
1. 판례번호 제거: {건수 또는 "해당 없음"}
2. 날짜 정규화: {처리 내역}
3. 꺽쇠 제거: {건수 또는 "해당 없음"}
4. 특수 띄어쓰기: {건수 또는 "해당 없음"}
5. 한자 병기 제거: {처리 내역}
6. 헤더/푸터 제거: {건수 또는 "해당 없음"}
7. 빈칸 처리: {처리 방식}
8. 과목별 특화: {적용 규칙명}
9. 정보 누락 없음: {확인 방법}
10. 자연스러운 문장: {QA 결과 요약}
```

## 규칙 적용 우선순위

```
1. 게시판 #462 피드백 (최신, 최우선)
2. R-09 (자의적 해석 금지) — 규칙 충돌 시 항상 우선
3. R-06, R-07 (목차 구조)
4. R-01~R-05, R-08 (텍스트 정제)
5. R-04 (호흡 조절) — 최종 단계
6. R-10, R-11 (검증) — 출력 후 필수
```

## 애매한 경우 처리

- R-09 위반이 의심되는 변환 → 원본 표현 유지
- 규칙 간 충돌 발생 시 → 오너(Opus)에게 판단 요청
- 파싱 품질이 낮아서 원문 확인 불가 시 → 사용자에게 PDF 직접 확인 요청
