# Context Guard — 대형 작업 자동 실행 모드

> 이 스킬이 호출되면 세션 종료까지 아래 규칙이 **자동 적용**됩니다.

## ⛔ 최우선 규칙 (모든 것보다 우선!)

**메인 에이전트는 Read/Edit/Write/Bash(빌드)를 직접 호출하지 않는다.**
**모든 작업은 `run_in_background=true` 서브에이전트로 디스패치한다.**

이유: 메인이 직접 작업하면 사용자 질문에 응답 불가.

---

## 작업 수신 시 필수 흐름 (매 요청마다 이 순서!)

```
1. 사용자 요청 접수
2. 작업 분해 (뭘 해야 하는지 정리)
3. 서브에이전트 백그라운드 디스패치 (아래 패턴 사용)
4. 사용자에게 "작업 시작했습니다" 1줄 응답
5. (사용자가 다른 질문하면 즉시 응답 가능)
6. 서브에이전트 완료 알림 → 결과 검수 → 보고
```

### 디스패치 패턴 (복붙해서 사용!)

**코드 구현/수정 (Opus):**
```
Agent(
  model="opus",
  subagent_type="general-purpose",
  run_in_background=true,
  description="코드 구현: {작업 요약}",
  prompt="목표: {한줄 설명}\n\n작업:\n1. {구체적 파일/수정 내용}\n2. ...\n\n커밋: [근형] {메시지}, Co-Authored-By 금지\npush: 하지 말 것"
)
```

**빌드/테스트 검증 (Sonnet):**
```
Agent(
  model="sonnet",
  subagent_type="general-purpose",
  run_in_background=true,
  description="빌드 검증: {작업 요약}",
  prompt="목표: {빌드/테스트 실행}\n\n명령어: {실행할 커맨드}\n결과만 요약해서 반환"
)
```

**탐색/게시판 (Haiku):**
```
Agent(
  model="haiku",
  subagent_type="general-purpose",
  run_in_background=true,
  description="탐색: {작업 요약}",
  prompt="목표: {검색/게시판 작업}\n\n결과만 요약해서 반환"
)
```

### 의존 관계 있는 작업

```
1단계: Agent(run_in_background=true, ...) → 완료 알림 대기
2단계: 1단계 결과 확인 후 → Agent(run_in_background=true, ...) 디스패치
```

포그라운드(`run_in_background` 없이)는 **절대 사용 금지**.
의존 관계가 있어도 백그라운드 → 완료 알림 → 다음 백그라운드 순서로 처리.

---

## 메인이 직접 해도 되는 것 (예외 목록)

- `git status`, `git branch` 같은 **1줄 상태 확인** (Bash)
- 사용자 질문에 **텍스트 응답**
- 서브에이전트 **결과 검수** (반환된 텍스트 읽기)
- 게시판 **읽기** (read_post, list_posts)

이 외의 모든 도구 호출 → 서브에이전트에 위임.

---

## 역할 배분

| 역할 | 모델 | 용도 |
|------|------|------|
| **설계/판단/리뷰** | Opus | 아키텍처, 의사결정, 복잡한 분석 |
| **코드 구현/수정** | Opus | 파일 수정, 테스트 작성, 리팩토링 |
| **빌드/검증/보조** | Sonnet | 빌드, 테스트 실행, 단순 치환 |
| **탐색/게시판** | Haiku | 파일 검색, 게시판 글 작성 |

> Sonnet 구현 → 품질 미달 → 리젝 반복 → 사용자 고통.
> 코드는 Opus. 한번에 제대로 하는 게 리젝 3번보다 싸다.

---

## 서브에이전트 프롬프트 필수 포함 항목

1. **목표**: 한 줄로 뭘 해야 하는지
2. **구체적 작업**: 파일 경로, 수정 내용, 주의사항
3. **커밋 규칙**: `[근형]` 형식, Co-Authored-By 금지
4. **push 여부**: 보통 "커밋만, push는 하지 말 것"
5. **워크트리**: 코드 작업이면 워크트리 생성 지시 포함

---

## 품질 관리

### 검수 체크리스트 (서브에이전트 완료마다)
1. 의도한 변경인가 — 요청 범위 벗어나지 않았는지
2. 부작용 없는가 — 다른 파일/기능에 영향 없는지
3. 빌드/테스트 통과 — 해당되면 Sonnet으로 검증 디스패치
4. 보고 퀄리티 — 사용자에게 보여줘도 되는 수준인지

### 품질 미달 시
- **resume**로 서브에이전트에 구체적 피드백 + 재작업 지시
- 같은 실수 반복 → 프롬프트 더 구체적으로 재작성
- Sonnet 보조 실패 → Opus로 대체

---

## 게시판 보고

- 작업 시작 → `lawear-work` 게시판에 플랜 글 (Haiku 위임)
- 작업 완료 → 댓글 또는 새 글 (Haiku 위임)
- 사용자에게는 링크만: `http://127.0.0.1:8585/posts/{id}`
- 300자 이하면 콘솔 직접 보고 OK
- **게시판 글 작성은 반드시 Haiku 서브에이전트 위임** (메인 직접 작성 금지)

---

## 사용자 인터럽트

- 사용자가 별 말 없으면 = 승인 → 계속 진행
- **메인이 사용자에게 묻지 않는다** — 게시판에 보고하고 진행
- 예외: 불가역 작업(push, 삭제)은 게시판에 "push 예정" 남기고 진행

---

## 컨텍스트 보호

| 수위 | 대응 |
|------|------|
| ~60% | 정상 운영 |
| 60~70% | `/compact` 고려 |
| 70~85% | 메인은 판단/지시만 |
| 85%+ | 메인은 검수+보고만, 전부 서브에이전트 |

서브에이전트 3개 이상 → 결과를 `docs/{작업명}/`에 파일 저장, 메인은 summary만 읽기.
