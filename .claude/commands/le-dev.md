## 팀 구조

### 메인 에이전트 (PM + TTS 변환)
- 사용자 요청 접수 + 티어 분류
- 사용자 피드백 즉시 응답
- **TTS 변환(Lv.1/2/3)은 메인이 직접 수행** — 규칙/기준 게시글을 직접 읽은 컨텍스트로 변환
- **코드 수정만 서브에이전트 위임** (dev-team 워크플로우)

### QA 역할 (메인 에이전트 내 순차 실행)

| 단계 | 역할 | 수행 |
|------|------|------|
| 1. 변환 | 작가 | 메인 에이전트가 직접. PDF 원문 + 로드된 규칙/기준 샘플 기반 |
| 2. 청취 QA | 수험생 | 메인 에이전트가 직접. 듣는 관점 자체 리뷰 |
| 3. 법리 QA | 판사 | 메인 에이전트가 직접. 원본 대비 누락/R-09 위반 검증 |
| 4. 최종 검증 | 사용자 | 게시판에 Lv.1/2/3 + diff 올려서 사용자 OK 대기 |

> 메인이 직접 하는 이유: 서브에이전트는 PM이 프롬프트에 넘긴 것만 본다.
> 메인은 규칙 로딩 스킬(/le-tts-load)로 기준 게시글/규칙을 직접 읽은 상태이므로 품질이 높다.

### 개발 워크플로우

이 프로젝트의 코드 수정/기능 추가/버그 수정은 `/dev-team` 공통 개발팀 워크플로우를 따른다.

**흐름:**
1. 사용자 요청 접수
2. 학습 오너가 전처리 전략/방향 결정

4. 품질 리뷰어가 TTS 체크리스트 10개 검증 (도메인 검증)
5. 학습 오너 최종 확인
6. 완료 보고

## TTS 변환 워크플로우

```
1. 요청 접수 (과목/파일 지정)
2. 티어 분류 (S/M/L/D)
3. 규칙 로딩 (/le-tts-load 호출 — 세션 최초 시 자동, 이미 로드됐으면 스킵)
   - tts_rules.md + 기준 게시글(#756,#760,#761) + 메모리 피드백 전부 Read
4. PDF 텍스트 추출 (서브에이전트 또는 Read 도구)
5. 메인 에이전트가 직접 TTS 변환 (Lv.1/2/3 동시)
   - 문제 → 목차 → 답안 순서
   - 호흡 조절 쉼표 삽입
   - 특수기호 최종 제거
6. diff 생성 (원본 vs TTS)
7. 자체 QA (수험생 관점 + 판사 관점)
8. 게시판에 Lv.1/2/3 + diff 올리기
9. 사용자 OK 대기 → OK 시 다음 문제
10. 커밋 (묶어서 또는 개별)
```

## 실행팀 필수 체크리스트 (10개)

작업 완료 시 아래 10개 항목 증빙 제출. 하나라도 빠지면 반려.

1. 판례번호 제거 — grep "판례번호 패턴" 결과 = 0건 첨부
2. 날짜 정규화 — 정규화 전후 샘플 3건 첨부
3. 꺽쇠 인용부호 제거 — grep "[<>《》]" 결과 = 0건 첨부
4. 특수 띄어쓰기 정리 — 정리 전후 diff 첨부
5. 한자 병기 제거 — grep 한자 패턴 결과 첨부
6. 페이지 헤더/푸터 제거 — 제거된 헤더/푸터 목록 첨부
7. 빈칸/○○○ 처리 — grep "○" 결과 = 0건 첨부
8. 과목별 특화 전처리 적용 — 적용된 규칙 목록 + 변환 샘플 첨부
9. 원본 대비 정보 누락 없음 — 원본 섹션 수 vs 변환 후 섹션 수 대조 첨부
10. TTS 자연스러움 확인 — 첫 3문장 + 마지막 3문장 샘플 첨부
11. 게시판에 작업 보고 — 게시판 글 ID + 링크 첨부

## 과목별 우선순위

```
1순위: 논술 5과목 (TTS 적합도 높음)
  민소 → 민법 → 형법 → 형소 → 부등법
2순위: 민사서류 (서술형 부분)
3순위: 부등서류 → 보류 (양식 처리 미정)
```
