# le-review — TTS 품질 리뷰

> 품질 리뷰어(Sonnet 서브에이전트)가 실행팀 결과물을 기계적으로 검증하는 스킬.
> 체크리스트 10개 자동 대조 — 하나라도 미제출/FAIL 시 즉시 반려.

## 역할

```
실행팀(Sonnet)이 TTS 변환 완료 후 체크리스트 10개 증빙을 제출하면,
이 스킬(품질 리뷰어)이 원본 PDF 텍스트와 대조하여 항목별 PASS/FAIL 판정.
최종 통과 시 오너(Opus)에게 전달.
```

## 전제 조건

- 입력 필수: 실행팀이 제출한 TTS 텍스트 + 체크리스트 10개 증빙 + 원본 PDF 텍스트
- 체크리스트 미제출 → 이하 검증 없이 즉시 반려
- 원본 PDF 텍스트 미제출 → "원본 없이 대조 불가" 반려

## 검증 워크플로우

```
1. 입력 수신
   - TTS 텍스트
   - 체크리스트 증빙 (10개)
   - 원본 PDF 텍스트

2. 체크리스트 미제출 여부 확인
   → 빠진 항목 있으면 즉시 반려

3. 원본 대조 (항목별 PASS/FAIL)
   → 아래 10개 검증 항목 순서대로

4. 총평 산출
   → ALL PASS → 통과 + 오너 전달 안내
   → FAIL 1개 이상 → 반려 + 구체적 사유 명시

5. 결과 출력
```

## 검증 항목 10개

| # | 항목 | 검증 방법 |
|---|------|----------|
| 1 | 판례번호 제거 완료 | TTS에 "(대판 ...", "(대결 ...", "다xxxxx" 패턴 잔존 여부 확인 |
| 2 | 날짜 정규화 완료 | "2020. 1. 5." 형태 잔존 여부, 상대적 시간 표현 적용 여부 (단, 시효/기간 계산 필수 날짜는 제외) |
| 3 | 꺽쇠 인용부호 제거 완료 | 「」『』 잔존 여부 확인 |
| 4 | 특수 띄어쓰기 정리 완료 | "근 거", "효 과" 등 비정상 띄어쓰기 잔존 여부 |
| 5 | 한자 병기 제거 완료 | 漢字 잔존 여부 (인명 甲乙丙은 갑/을/병으로 변환됐는지 확인) |
| 6 | 페이지 헤더/푸터 제거 완료 | "1/3 법무사...", 페이지 번호, 시험지 헤더 패턴 잔존 여부 |
| 7 | 빈칸/○○○ 처리 완료 | ○○○, □□□, 빈칸 표시 잔존 여부 (처리 방식 확인) |
| 8 | 과목별 특화 전처리 적용 완료 | 과목에 따른 특화 규칙 적용 여부 (아래 과목별 기준 참조) |
| 9 | 원본 대비 정보 누락 없음 확인 | 원본 텍스트의 논지/내용이 TTS에 유지됐는지 대조 |
| 10 | TTS로 읽었을 때 자연스러운 문장 확인 | 끊어 읽기 불자연스러운 부분, 법률 용어 전달 오해 가능성 여부 |

### 과목별 특화 전처리 기준 (항목 8 참조)

| 과목 | 확인 포인트 |
|------|------------|
| 민소/민법 | 빈칸 박스 처리 방식 일관성 |
| 형법/형소 | 별표(★★) → "중요" 변환 여부, 중첩 괄호 단순화 여부 |
| 부등법 | "명문x" → "명문 규정 없음" 변환 여부 |
| 민사서류 | 표/양식 → 줄글 재구성 여부 |
| 부등서류 | 보류 상태 — 해당 없음 |

## 결과 출력 형식

```
## 품질 리뷰 결과 — {과목}/{파일명}

| # | 항목 | 판정 | 사유 |
|---|------|------|------|
| 1 | 판례번호 제거 | ✅ PASS | 잔존 없음 |
| 2 | 날짜 정규화 | ❌ FAIL | 17행: "2020. 3. 5." 원문 형태 그대로 잔존 |
| 3 | 꺽쇠 인용부호 제거 | ✅ PASS | 잔존 없음 |
| ... | ... | ... | ... |

### 총평

[통과 / 반려]

통과 시: "10개 항목 ALL PASS. 오너(Opus) 최종 확인으로 전달."
반려 시: "FAIL X개. 사유 명시 항목 수정 후 재제출 요청."

### 반려 사유 상세 (FAIL 항목만)

- 항목 2: 17행 — 원문: "2020. 3. 5." → 미처리. "2020년 3월 5일" 또는 상대적 표현으로 변환 필요
- 항목 X: ...
```

## 판정 기준 상세

### PASS 조건
- 검증 항목에 해당하는 원본 패턴이 TTS에서 모두 처리됨
- 처리 방식이 tts_rules.md 규칙과 일치함

### FAIL 조건
- 원본에 해당 패턴이 있는데 TTS에서 미처리
- 처리했으나 규칙과 다른 방식으로 변환됨 (예: 판례번호를 자연어로 변환 — 이건 FAIL, 완전 제거해야 함)
- R-09 위반: TTS에 원본에 없는 내용이 추가됨

### 판단 어려운 경우
- 오너(Opus) 판단 필요 표시: `⚠️ 오너 확인 요청` 후 통과
- 리뷰어가 자의적으로 FAIL 처리하지 말 것 — 판단이 필요한 경우 오너에게 위임

## 반려 시 재제출 안내

```
반려 사유를 명확히 전달:
- 어느 행에서 어떤 패턴이 잔존/잘못 처리됐는지
- 올바른 처리 방식 (tts_rules.md 규칙 번호 포함)

재제출 시 수정된 항목만 재검증 (전체 재처리 불필요).
```

## 우선순위 규칙

- **게시판 #462 피드백 > tts_rules.md** — 피드백이 최신 규칙
- R-09(자의적 해석 금지)는 검증 기준에서도 최상위
- FAIL이 R-09 위반이면 다른 항목 PASS 여부와 관계없이 반려
