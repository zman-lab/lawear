# TTS 파이프라인 설계

## 전체 흐름

```
Phase 1          Phase 2           Phase 3           Phase 4
PDF 파싱    →    AI 전처리    →    TTS 텍스트 생성  →  TTS 엔진 호출
(PyMuPDF)        (규칙 적용)       (과목별 템플릿)      (음성 파일)
```

## Phase 1: PDF → 텍스트 추출

### 도구
- **PyMuPDF** (fitz) — Python 라이브러리

### 처리 내용
- PDF에서 텍스트 레이어 추출 (OCR 불필요 — 전 과목 텍스트 임베딩 확인 완료)
- 페이지 단위 텍스트 + 메타데이터
- 밑줄/강조 어노테이션 별도 추출 (어노테이션 레이어)

### 밑줄/강조 추출 방안
- PyMuPDF의 `page.annots()` 또는 `page.get_text("dict")`로 텍스트 스타일 정보 추출
- 밑줄(underline), 볼드(bold), 색상(color) 등 스타일 플래그 활용
- 추출된 강조 정보를 Phase 2에서 "중요" 마커로 활용 가능
- 현재 미검증 — Phase 1 구현 시 실제 PDF로 테스트 필요

### 출력
- 과목/파일별 원본 텍스트 파일
- (선택) 어노테이션 메타데이터 JSON

## Phase 2: AI 전처리

### 도구
- LLM API (Anthropic Claude 등)

### 처리 내용
- `tts_rules.md`의 기본 규칙 10개 + 추가 규칙 11개 적용
- 과목별 특화 전처리 적용
- AI 자의적 해석 금지 (R-09) — 원본 정보만 사용

### 입력
- Phase 1에서 추출한 원본 텍스트
- 전처리 규칙 세트 (프롬프트에 포함)
- 과목 식별 정보

### 출력
- 전처리 완료 텍스트
- diff (원본 vs 전처리)
- 체크리스트 증빙

### 과목별 처리 분기
```
논술 5과목 → 표준 전처리 + 과목 특화 규칙
민사서류   → 표 데이터 줄글 재구성 + 표준 전처리
부등서류   → 보류 (양식 처리 미정)
```

## Phase 3: TTS 텍스트 생성

### 처리 내용
- 표준 출력 형식 적용 (`sample_format.md` 참조)
- 문제 → 목차 → 답안 순서 구성
- 호흡 조절 쉼표 최종 삽입
- 특수기호 최종 제거 확인

### 출력
- TTS 입력용 최종 텍스트 파일
- diff (Phase 2 결과 vs Phase 3 결과)
- QA 결과

## Phase 4: TTS 엔진 호출

### 처리 내용
- TTS 엔진에 텍스트 전달
- 음성 파일(mp3/wav) 생성
- 파일명 규칙: `{과목}_{파일번호}_{문제번호}.mp3`

### TTS 엔진 후보 (미확정)
- Google Cloud TTS
- CLOVA Voice (네이버)
- OpenAI TTS
- Edge TTS (무료)
- 기타 한국어 지원 엔진

### 출력
- 음성 파일
- 재생 시간 메타데이터

## 배치 처리 설계

### 단위
- 1 배치 = 1 PDF 파일의 전체 문제/답안

### 흐름
```
590개 PDF
  → 과목별 분류 (7개 그룹)
  → 과목 내 파일별 순차 처리
  → 파일 내 문제별 병렬 처리 가능
```

### 우선순위
1. 논술 5과목 (TTS 적합도 높음, 즉시 활용 가능)
2. 민사서류 (서술형 부분 우선)
3. 부등서류 (보류 → 추후 처리)

## 자동화 예상치
- 논술 5과목 + 민사서류 서술 부분: 90%+ 자동화
- 민사서류 표 데이터: 70~80% 자동화 (AI 줄글 재구성 + 수동 검수)
- 부등서류 양식: 보류
