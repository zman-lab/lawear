# TTS 전처리 규칙 (확정판)

사용자(수험생) 피드백 2차례 반영 완료. 이 문서가 TTS 변환의 최종 규칙.

## 기본 전처리 규칙 (10개)

| # | 규칙 | 예시 |
|---|------|------|
| 1 | 한자 인명 → 한글 | 甲→갑, 乙→을, 丙→병 |
| 2 | **판례번호 완전 제거** | ~~(대판 2018.7.19. 2018다22008)~~ → 삭제 |
| 3 | 날짜 정규화 | 2020. 1. 5. → 2020년 1월 5일 |
| 4 | 페이지 헤더/푸터 제거 | "1/3 법무사형법..." 등 삭제 |
| 5 | 꺽쇠 인용부호 제거 | 「부동산등기법」→ 부동산등기법 |
| 6 | 특수 띄어쓰기 정리 | "근 거" → "근거" |
| 7 | 한자 병기 제거 | 구법(舊法) → 구법 (인명 甲乙丙은 한글 변환) |
| 8 | 원문자 번호 통일 | ① → 첫째 또는 1번 |
| 9 | O/x 표기 변환 | 경우-O → 인정됨, 경우-x → 부정됨 |
| 10 | 빈칸(○○○) 처리 | ○○○원 → (금액)원 또는 정답 삽입 |

## 사용자 피드백 반영 추가 규칙 (11개)

### R-01. 판례번호는 변환이 아닌 완전 제거
- **기존**: 판례번호를 자연어로 변환 (대판 2018.7.19. → 대법원 2018년 7월 19일 판결)
- **확정**: 판례번호 자체를 완전히 제거. TTS에서 불필요한 정보.
- 예: "(대판 2018.7.19. 2018다22008)" → 삭제

### R-02. 날짜 → 상대적 시간 표현 활용
- 날짜가 빈번하면 듣는 사람이 날짜 파악이 불가능
- 문맥상 가능하면 상대적 표현으로 변환
- 예: "2000년 1월 1일 대여, 2005년 3월 소장접수" → "2000년 1월경 대여하고 5년이 지난 이후 소장 접수"
- 단, 시효/기간 계산에 필수인 날짜는 유지

### R-03. 괄호(A(B)) → "A 혹은 B"
- TTS가 괄호를 읽지 못함
- 예: "행위시법주의(소급효금지의 원칙)" → "행위시법주의 혹은 소급효금지의 원칙"
- 보충 설명 괄호도 동일: "채무자(피고)" → "채무자 혹은 피고"

### R-04. 호흡 조절 쉼표 삽입
- 긴 문장은 문맥에 맞게 쉼표(,)를 삽입하여 TTS가 끊어 읽도록 유도
- TTS 엔진은 쉼표(,)와 마침표(.)에서 호흡을 쉼
- 너무 긴 한 문장 → 듣는 사람이 암기 불가
- 반대로 불필요한 위치에 쉼표 남발 → 끊겨 들림
- 듣는 사람의 암기와 이해를 기준으로 판단

### R-05. 콜론(:) → 쉼표(,)
- TTS가 콜론을 읽지 못함
- 예: "결론: 타당하지 않다" → "결론, 타당하지 않다"
- 예: "효과: 증명책임이 전환된다" → "효과, 증명책임이 전환된다"

### R-06. 문제 후 목차 먼저 → 답안
- 문제를 읽은 직후 답안의 목차를 먼저 한번 읽어줌
- 그 후 목차 순서에 맞게 답안을 구성하여 읽음
- 이유: 판례를 못 외워도 목차만큼은 꼭 외워야 함

### R-07. 목차는 원문 그대로
- 목차를 요약하거나 재구성하지 않음
- PDF 원본의 목차 표현을 그대로 사용

### R-08. 특수기호 제거
- TTS가 읽지 못하는 기호들을 제거 또는 변환
- 대상: `[]` `—` `:` `()` `/`
- 대괄호 안 내용은 문맥에 맞게 자연어로 변환
- 예: "[문제]" → "문제" 또는 구분자로 활용

### R-09. AI 자의적 해석 금지 — 원본 정보만 사용
- AI가 법률 지식으로 내용을 보충/해석/변경하면 안 됨
- PDF에 있는 정보만으로 TTS 텍스트 구성
- "TTS 맥락 자연스러움"과 "AI 자의적 해석"의 경계를 명확히
- 의심스러운 경우 원본 표현 유지가 원칙

### R-10. diff 필수 제공
- 매 변환 결과에 원본 vs TTS 비교(diff)를 함께 제공
- 사용자가 최종 검수할 수 있도록

### R-11. 듣는 사람 입장 QA 필수
- 변환 완료 후 아래 관점에서 자체 QA 수행:
  - 이 부분이 구분되어 들리는가?
  - 문장이 너무 길어서 맥락 이해나 암기가 힘들지 않은가?
  - 날짜/인명이 혼동되지 않는가?
  - 법률 용어가 정확히 전달되는가?

### R-12. 문제 부분 간결화 (2026-03-14 피드백)
- 문제(problem) 섹션의 TTS 텍스트가 너무 길고 반복되는 단어/표현이 많음
- 쉼표를 과도하게 넣지 말 것 — TTS가 쉼표마다 쉬면서 읽어 너무 느려짐
- 원문의 핵심 정보는 유지하되, 중복 표현/불필요한 수식어는 제거
- "자의적 해석 금지" 규칙(R-09)과 균형: 내용 변경은 금지하되 표현의 간결화는 허용

### R-13. 목차에서 상투적 항목 제거 (2026-03-14 피드백)
- 목차(toc) 섹션에서 "결론"과 "이유"는 거의 모든 문제에 등장하는 상투적 항목
- 목차에 "결론", "이유" 단독 항목은 제거 (읽어줄 필요 없음)
- 단, "결론의 당부" 등 구체적 내용이 붙은 경우는 유지

## 과목별 특화 전처리

| 과목 | 특화 규칙 |
|------|----------|
| 민소/민법 | 빈칸 박스 처리 전략 (빈칸 유지 vs 정답 삽입 — 선택 필요) |
| 형법/형소 | 별표(★★) → "중요" 변환, 중첩 괄호 단순화 |
| 부등법 | "명문x" → "명문 규정 없음" 변환 |
| 민사서류 | 등기부/가족관계 표 → AI 줄글 재구성 |
| 부등서류 | 보류 (등기신청서 양식 처리 미정) |

### R-14. 제+숫자 복합어 TTS 읽기 (2026-03-15 피드백)
- TTS가 "제1심", "제3자" 같은 한자복합어+숫자 패턴을 못 읽음
- 제1심 → 제일심, 제2심 → 제이심, 제3자 → 제삼자
- 법조문 번호(제44조, 제216조 등)는 TTS가 정상 처리하므로 그대로 유지

### R-15. 숫자 한글화 + 불필요 수식어 과감 생략 (2026-03-15 피드백)
- 문제 섹션에서 답안 구성과 무관한 수식어 과감 제거
- "현금 1000만원을 직접 건네며 빌려주면서" → "천만원을 빌려주며"
- 금액 한글화: 1000만원 → 천만원, 1억원 → 일억원 (TTS가 더 자연스러움)
- "현금", "직접 건네며" 등 꾸미는 말은 듣기 방해

### R-16. 조문 인용 시 제목 삽입 + 조문 앞배치 (2026-03-15 피드백)
- 조문 번호만 나열하지 말고 제목을 함께 읽어줄 것
- "제397조" → "제397조 금전채무불이행에 대한 특칙에 의해"
- 조문이 설명 뒤에 나오면 비효율 → 설명 앞에 배치
- 법령 API로 조문 제목 자동 조회 활용 (가능한 경우)

### R-17. 문맥 유추 가능한 정보 생략 (2026-03-15 피드백)
- 당사자가 두 명뿐이고 문맥상 자연히 유추되면 이름 반복 줄이기
- 구체 날짜보다 상대 표현이 간결하면 대체: "변제일 다음날부터"
- "갑이 현재 을을 상대로" → 문맥상 생략 가능하면 생략

### R-18. diff 누락 시 작업 미완료 (2026-03-15 피드백)
- R-10(diff 필수 제공) 재강조
- 원본 vs TTS 변환 비교를 **반드시** 포함
- diff가 없으면 사용자가 검증 불가 → 작업 미완료 취급

### R-19. 조문 번호는 아라비아 숫자 그대로 (2026-03-15 피드백)
- "형법 이백오십조" X → "형법 제250조" O
- 민법/민소에서 이미 "제397조" 식으로 하고 있으므로 전 과목 통일
- "제250조 제1항 제5호" → 그대로 OK
- 조문 번호를 한글 서수로 변환하지 말 것

### R-20. 목록 번호는 숫자로 (2026-03-15 피드백)
- "첫째, 둘째, 셋째" X → "1. 2. 3." O
- TTS가 숫자 목록 번호는 정상적으로 읽음
- 로마 숫자(Ⅲ, Ⅳ)만 못 읽으므로 아라비아 숫자로 변환
- "하나, 둘, 셋" 같은 한글 서수도 사용하지 말 것

### R-21. 숫자 한글화 세부 규칙 (2026-03-15 피드백)
- 금액: 한글화 O (1000만원 → 천만원, 6억원 → 육억원)
- 조문 번호: 한글화 X (제250조 그대로)
- 목록 번호: 한글화 X (1. 2. 3. 그대로)
- 호실: 자연스러운 한글 읽기 (302호실 → 삼백이호실, "삼공이" X)
- 주소: 자연스러운 한글 읽기 (365번지 → 삼백육십오번지)
- 기타 숫자: 사건의 쟁점이 되어 답안에 현출된 경우에만 기재
- 쟁점과 무관한 숫자(날짜 등)는 R-17에 따라 생략 가능

### R-22. 모든 피드백은 전 과목 공통 (2026-03-15 피드백)
- 사용자가 특정 과목 게시글에 댓글로 준 피드백이라도 전 과목에 동일 적용
- 특정 과목에만 적용되는 피드백이면 사용자가 "이건 형법에만 적용" 식으로 명시함
- 명시가 없으면 기본적으로 모든 과목 공통

## TTS 변환 3인 팀 구조 (2026-03-22 갱신)

TTS 변환은 **메인 에이전트가 직접** 아래 단계를 순차 실행.
서브에이전트는 규칙/기준 게시글 컨텍스트가 없으므로 TTS 변환에 부적합.

### Phase 1: 작가 (문장 다듬기)
- PDF 원본 추출 → TTS 규칙(R-01~R-14) 적용
- 문제: 대폭 간결화 (조사 최소화, 구어체, 필러 제거, 질문 간소화)
- 답안: 번호 체계(1. 2. 3.), "~이다," 종결, "생각건대" 제거
- 목차: "결론"/"이유"/"사안의 해결" 제거, 핵심 쟁점만
- 출력: 원본 + TTS 변환 텍스트

### Phase 1.5: 증빙 (기계적 검증)
- 체크리스트 10개 증빙 — 기본 전처리 적용 확인
- R-01~R-23 적용 증빙 — 각 규칙별 적용 여부 (해당 없으면 N/A)

### Phase 2: 수험생 리뷰어 (TTS 청취자 관점)
- Phase 1 결과를 "귀로 듣는 사람" 입장에서 검토
- 체크포인트:
  - 호흡/끊어읽기 자연스러운가?
  - 문장이 너무 길어서 암기 불가능하지 않은가?
  - 날짜/인명(갑을병) 혼동되지 않는가?
  - 쉼표 위치 적절한가? (과다/부족)
  - 한번 듣고 목차 구조가 머리에 들어오는가?
- 피드백 → Phase 1 결과 수정

### Phase 3: 판사 QA (채점자 관점)
- 원본 PDF 대비 중요 법리/키워드 누락 여부 확인
- 시험 채점 기준에서:
  - 핵심 쟁점이 빠진 게 없는가?
  - 결론이 정확한가?
  - 법리 적용이 올바른가?
- R-09 위반 여부 (AI 자의적 해석이 들어갔는가?)
- 최종 판정: PASS / 수정필요 (사유 명시)

### 팀 운영 규칙
- **메인 에이전트가 직접** 순차 실행 (서브에이전트 위임 금지)
- 이유: 메인이 /le-tts-load로 규칙+기준 샘플을 직접 읽은 상태이므로 품질 보장
- Phase 2에서 수정 발생 시 Phase 1으로 돌아가지 않고 직접 수정
- Phase 3에서 PASS 나올 때까지 반복하지 않음 (1회 QA → 수정필요 시 사유만 표기)

### R-23. Lv.1~3 변환 시 문제별 사용자 확인 (2026-03-22 피드백)
- 전과목 자동 일괄 변환 금지
- Case 1개의 문제 1개를 Lv.1/Lv.2/Lv.3 한꺼번에 변환 → diff 비교표로 사용자에게 제시
- 사용자 확인 → 다음 문제. 시간이 걸려도 품질 우선 (법률 시험은 한 글자 차이로 오답)
- 사용자가 "묶어서 해도 돼"라고 명시하기 전까지 1개씩
- 사용자 피드백은 다음 문제에 즉시 반영
- 비교표 형식 기준: 게시판 #756 (형소 제1문 확정판)
- 각 레벨 전문을 생략 없이 제시 (... 으로 줄이지 말 것)
- Lv.1/2/3 답안 전문 + diff 요약표 필수

## 규칙 적용 우선순위

1. R-09 (자의적 해석 금지) — 최상위. 다른 규칙과 충돌 시 이 규칙이 우선
2. R-06, R-07 (목차 구조) — 답안 구성의 뼈대
3. R-13 (목차 상투적 항목 제거) — 목차 정리 단계에서 적용
4. R-01~R-05, R-08, R-14, R-19~R-21 (텍스트 정제) — 기계적 적용
5. R-04 (호흡 조절), R-12 (문제 간결화) — 최종 단계에서 듣기 관점 조정
6. R-10, R-11 (검증) — 출력 후 필수
7. 3인 팀 구조 — 모든 TTS 변환 작업에 적용
8. R-23 (문제별 사용자 확인) — Lv.1~3 변환 프로세스 규칙
