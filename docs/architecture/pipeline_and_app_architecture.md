### 1.3 각 Phase 상세

#### Phase 1: PDF 텍스트 추출

| 항목 | 내용 |
|------|------|
| 도구 | PyMuPDF (fitz) |
| 입력 | ~/Downloads/2026_USB/ (590 PDF) |
| 출력 | raw/{과목}/{파일}.txt |
| 자동화 | 100% (기계적 추출) |
| 소요 | ~5초/PDF, 전체 ~50분 |

```python
# 핵심 로직
import fitz

def extract_pdf(pdf_path: str) -> str:
    doc = fitz.open(pdf_path)
    text = ""
    for page in doc:
        text += page.get_text("text")
    return text
```

**1차 필터링**: 590개 중 TTS 대상(미케/모고/답안) 251개만 처리. 강의계획서, 쪽지시험 등은 제외.

#### Phase 2: AI TTS 전처리

| 항목 | 내용 |
|------|------|
| 도구 | Claude Opus (3인팀: 작가-수험생-판사) |
| 입력 | raw/ 텍스트 + TTS 규칙 R-01~R-18 |
| 출력 | tts/{과목}/{파일}/{case}.txt |
| 자동화 | 반자동 (Claude 세션에서 수동 실행) |
| 소요 | ~2분/문제, 전체 ~30시간 (900문제) |

**기계적 전처리 (rules.py로 자동화 가능)**:
- R-01: 한자 인명 변환 (甲→갑, 乙→을)
- R-02: 판례번호 제거 (정규식)
- R-03: 날짜 정규화 (2020. 1. 5. → 2020년 1월 5일)
- R-04: 헤더/푸터 제거 (페이지 번호 패턴)
- R-05: 꺽쇠 인용부호 제거
- R-06: 특수 띄어쓰기 정리
- R-07: 한자 병기 제거
- R-14: 제+숫자 복합어 (제1심→제일심)
- R-16: 조문 제목 자동 삽입 (법령 API 캐시 활용)

**AI 판단 필요 (Opus 3인팀)**:
- R-02/R-17: 날짜→상대적 시간, 문맥 생략
- R-03: 괄호 변환 (보충설명 vs 동의어 판별)
- R-04/R-12: 호흡 쉼표 삽입, 문제 간결화
- R-15: 불필요 수식어 제거

#### 오프라인 전략

```
[옵션 A] PWA + Service Worker (권장)
+-------------------------------------------+
| PWA (Progressive Web App)                 |
|                                            |
| Service Worker                             |
|   ├── 앱 셸 캐시 (HTML/CSS/JS)           |
|   └── 오디오 캐시 (선택적 다운로드)       |
|       ├── 과목 단위 다운로드 버튼         |

+-------------------------------------------+
  장점: 별도 앱스토어 불필요, URL 공유, 즉시 배포
  단점: iOS Safari 캐시 제한 (50MB), 백그라운드 재생 불안정

[옵션 B] Capacitor (네이티브 래핑) — 추후 필요 시
+-------------------------------------------+
| Capacitor 앱                              |
|   ├── 기존 React 코드 그대로              |
|   ├── 파일시스템에 MP3 저장               |
|   └── 네이티브 오디오 플레이어            |
+-------------------------------------------+
  장점: 캐시 제한 없음, 백그라운드 재생 안정
  단점: 앱스토어 배포, 빌드 복잡도 증가
```

**권장**: PWA 먼저 → 부족하면 Capacitor 추가. 현재 플레이어(배속, 반복, 슬립타이머)는 두 옵션 모두 호환.

### 1.7 기술 스택

| 영역 | 선택 | 이유 |

### 2.1 TTS 엔진 비교

| 엔진 | 한국어 품질 | 법률 용어 | 배속 음질 | 가격 (100만자) | 무료 | 비고 |

| **OpenAI TTS** | B+ | B | A | $15 | 없음 | 한국어는 아직 부자연스러움 |

- 이유: 한국어 발음 정확도 최상급, 법률 용어 처리 양호, 무료 할당량으로 1차 배치 커버
- 차선: Edge TTS (무료, 품질은 타협)

### 2.2 한국어 법률 용어 발음 이슈

| 문제 | 예시 | 대응 |
|------|------|------|
| 한자어 장음 | 판결(判決) vs 판결(板結) | TTS 엔진이 문맥 판단 — 대부분 정확 |
| 법률 전문용어 | 기판력, 소송물, 변론종결 | Google/Naver 모두 정확 처리 |
| 숫자+한자 복합어 | 제1심, 제3자 | R-14로 사전 변환 (제일심, 제삼자) |
| 약어 | 민소법, 부등법 | TTS 텍스트에서 풀어쓰기 |
| 길고 복잡한 합성어 | 소유권이전등기말소청구소송 | 쉼표로 끊어읽기 유도 |

**실전 테스트 필수**: 엔진 확정 전 샘플 10문제를 4개 엔진으로 렌더링 → 블라인드 비교 청취.

### 2.3 배속 변경 전략

```
원본 렌더링: 1.0x (Cloud TTS 기본 속도)
앱에서 조절: HTML5 Audio playbackRate (0.5x ~ 3.0x)
```

| 방식 | 음질 | 구현 | 권장 |
|------|------|------|------|
| API speed 파라미터 | 좋음 (엔진 내부 처리) | 배속별 MP3 별도 생성 | X (저장 공간 5배) |
| playbackRate | 2x까지 양호, 3x 약간 열화 | 코드 1줄 | O (기존 방식 유지) |
| 시간축 압축(WSOLA) | 우수 | ffmpeg/rubberband | 필요시 후처리 |

**권장**: `playbackRate` 그대로 유지. 3x 이상에서 열화 발생 시 WSOLA 후처리 파이프라인 추가.

### 2.4 청크 단위 결정

```
[권장] 문제 전체 = 1 MP3 파일

이유:
  - 문장별 → 파일 수 폭증 (900 x 30문장 = 27,000개), 관리 불가
  - 문단별 → 문제/목차/답안 경계가 불명확
  - 문제 전체 → 1 Case = 1 파일, 직관적, 탐색/반복 용이

트레이드오프:
  - 문제 전체가 길면 (10분+) 중간 탐색이 불편
  - 대안: 메타데이터에 섹션 타임스탬프 포함

  {
    "case01_q01": {
      "duration": 390,
      "sections": {
        "problem": {"start": 0, "end": 45},
        "toc": {"start": 45, "end": 62},
        "answer": {"start": 62, "end": 390}
      }
    }
  }
  → 앱에서 "문제만 다시", "답안부터" 탐색 가능
```

### 2.5 무음 구간 삽입 전략

| 위치 | 무음 길이 | 이유 |
|------|----------|------|
| 문제 → 목차 전환 | 1.5초 | "지금부터 목차입니다" 인지 |
| 목차 → 답안 전환 | 2.0초 | 목차 암기 시간 확보 |
| 목차 항목 사이 | 0.5초 | 항목 구분 |
| 답안 대항목 전환 | 1.0초 | 논리 전환 인지 |
| 문장 내 쉼표 | TTS 엔진 자동 | 별도 처리 불필요 |

**구현**: TTS 텍스트에 SSML `<break time="1500ms"/>` 삽입, 또는 렌더링 후 ffmpeg로 무음 삽입.

## Phase 3: 수험생 사용자 리뷰

### 3.1 사용 시나리오

```
[시나리오 1] 통학 중 (지하철 30분)
  → 한 과목 미케01 전체 듣기 (6문제, ~45분)
  → 2배속이면 ~23분
  → 오프라인 필수 (지하철 터널)

[시나리오 2] 운동 중 (러닝 40분)
  → 형법 문제 셔플 모드
  → 블루투스 이어폰, 화면 안 봄
  → 자동 다음 문제 전환

[시나리오 3] 취침 전 (30분)
  → 슬립 타이머 30분
  → 부등법 모고01 반복
  → 화면 밝기 최소 (다크 모드)

[시나리오 4] 책상 공부 (2시간)
  → 민소 미케01 Case2 설문3만 반복
  → 1.0x로 천천히
  → 텍스트 보면서 따라읽기 (가사 모드)
```

### 3.2 과목 전환 UX

```
현재: 과목 선택 → 파일 선택 → 문제 선택 (3단계)
개선:
  +------+------+------+------+------+
  | 민소 | 민법 | 형법 | 형소 | 부등 |  ← 탭 전환 (스와이프)
  +------+------+------+------+------+
  | [미케01]                         |
  |   Case 1/설문1  ▶ 6:30          |
  |   Case 1/설문2    7:15          |
  |   Case 2/설문1    8:40          |
  | [미케02]                         |
  |   ...                            |
  +----------------------------------+
  |  ◀◀  ▶/❚❚  ▶▶  🔁  ⏱️  1.5x   |  ← 하단 고정 미니 플레이어
  +----------------------------------+

  → 과목 탭 전환해도 현재 재생 유지
  → 다른 과목 문제 선택 시 자동 전환
```

### 3.3 "이 문제 다시 들려줘" UX

```
[현재 구현됨]
  - repeat-one 모드: 한 문제 무한 반복 ✓
  - 배속 조절: 0.5x ~ 3.0x ✓

[추가 필요]
  - 섹션 탐색: "문제만", "목차만", "답안만" 재생
    → MP3 메타데이터의 섹션 타임스탬프 활용
  - 10초 뒤로: 놓친 부분 다시 듣기 버튼
  - 북마크: "이 문제 다시" 표시 → 북마크 리스트에서 모아 듣기
```

### 3.4 다운로드 용량 추정

```
1 문제 = ~1,620자 → TTS 음성 ~4~7분 (1.0x 기준)
MP3 128kbps: 1분 = ~1MB

900문제 x 평균 5.5분 = 4,950분 = ~4.8GB (전체)

과목별:
  민법:     250문제 x 5.5분 = ~1.3GB
  형법:     200문제 x 5.5분 = ~1.1GB
  민소:     180문제 x 5.5분 = ~1.0GB
  부등:     170문제 x 5.5분 = ~0.9GB
  형소:     100문제 x 5.5분 = ~0.5GB
```

| 전략 | 용량 | 장단점 |
|------|------|--------|
| 전체 다운로드 | ~4.8GB | 완전 오프라인, 초기 다운로드 부담 |
| 과목별 다운로드 | ~1GB/과목 | 선택적, 권장 |
| 파일별 다운로드 | ~30MB/파일 | 세밀, UI 복잡 |
| 스트리밍 + 캐시 | 0 (초기) | 네트워크 필요, 들은 것만 캐시 |

**권장**: 과목별 다운로드 + 스트리밍 캐시 병행.
- 기본: 스트리밍 (WiFi에서 자동 캐시)
- 수동: "민소 전체 다운로드" 버튼 → Service Worker가 과목 MP3 일괄 캐시
- 용량 표시: "민소 다운로드 (1.0GB)" UI에 명시

### 3.5 배터리 소모 비교

| 방식 | CPU 사용 | 배터리 | 비고 |
|------|---------|--------|------|
| 브라우저 SpeechSynthesis | 높음 (실시간 합성) | 많이 소모 | 1시간 ~15% |
| MP3 재생 (하드웨어 디코딩) | 극히 낮음 | 최소 | 1시간 ~3% |
| MP3 + 화면 꺼짐 | 극히 낮음 | 최소 | 1시간 ~2% |

**결론**: MP3 프로덕션 모드는 배터리 소모 5~7배 절감.

### 3.6 MP3 비트레이트 트레이드오프

| 비트레이트 | 음질 | 용량 (900문제) | 권장 |
|-----------|------|-------------|------|
| 64kbps | 충분 (음성) | ~2.4GB | 용량 우선 시 |
| 96kbps | 좋음 | ~3.6GB | 균형 |
| 128kbps | 우수 | ~4.8GB | 품질 우선 시 |

**권장**: 64kbps. 음성 전용이므로 64kbps로도 충분한 품질. 용량 절반으로 오프라인 부담 감소.

→ 64kbps 기준 재계산:
- 전체: ~2.4GB
- 과목별: ~500MB
- 파일별: ~15MB

### 대안 비교 (Edge TTS vs Google Cloud)
