# 팀 구조 — 이중 검증 체계

## 배경

초기 설계는 오너(Opus)가 전략+감독+최종확인을 모두 담당했으나, 사용자 피드백으로 "오너가 누락할 가능성"이 지적됨. 토큰을 더 쓰더라도 정확성이 우선이므로 이중 검증 체계로 전환.

## 조직도

```
학습 오너 (Opus) — 전략 + 최종 확인
    |
품질 리뷰어 (Sonnet 서브에이전트) — 1차 기계적 검증
    |
실행팀 — Sonnet(구현) + Haiku(탐색/보고)
```

## 각 역할 상세

### 학습 오너 (Opus)
- 과목별 전처리 전략 설계/판단
- 최종 품질 확인 (리뷰어가 놓친 것만 잡기)
- "자연스러움" 등 판단이 필요한 부분 집중
- L/D 티어 의사결정
- 학습 우선순위/스케줄 조율
- 전략 수정이 필요한 패턴 발견 시 규칙 업데이트

### 품질 리뷰어 (Sonnet 서브에이전트)
- 실행팀이 제출한 체크리스트 10개 자동 대조
- 전처리 규칙 적용 여부 기계적 검증
- 원본 PDF 대비 누락/왜곡 검출
- 체크리스트 미제출 → 즉시 반려

### 실행팀
- **Sonnet**: PDF 텍스트 추출, AI 전처리, 양식→줄글 재구성, TTS 텍스트 생성
- **Haiku**: 파일 탐색, 진행 상황 게시판 보고, 단순 정규식 치환
- 작업 완료 시 체크리스트 증빙 강제 제출

## 이중 검증 흐름

```
실행팀(Sonnet)
  → 작업 완료 + 체크리스트 10개 증빙 제출
      ↓
리뷰어(Sonnet)
  → 1차 검증 (체크리스트 자동 대조, 기계적 확인)
  → 미통과 시 반려 + 사유 명시
      ↓
오너(Opus)
  → 최종 확인 (판단 필요한 부분만)
  → 자연스러움, 맥락 적절성, AI 자의적 해석 여부
```

## 실행팀 필수 제출 체크리스트 (10개)

작업 완료 시 아래 10개 항목 증빙을 반드시 제출. 하나라도 빠지면 반려.

1. 판례번호 제거 완료
2. 날짜 정규화 완료 (상대적 시간 표현 포함)
3. 꺽쇠 인용부호 제거 완료
4. 특수 띄어쓰기 정리 완료
5. 한자 병기 제거 완료 (인명 甲乙丙은 한글 변환)
6. 페이지 헤더/푸터 제거 완료
7. 빈칸/○○○ 처리 완료
8. 과목별 특화 전처리 적용 완료
9. 원본 대비 정보 누락 없음 확인
10. TTS로 읽었을 때 자연스러운 문장 확인

## 오너 감독 범위 (최종 확인에 집중)

- 리뷰어가 통과시킨 결과물의 "듣기 자연스러움" 판단
- 법률 용어/맥락이 TTS에서 오해를 유발하지 않는지
- 과목 간 전처리 일관성 확인
- AI 자의적 해석이 개입되지 않았는지 (R-09)
- 전략 수정이 필요한 패턴 발견 시 규칙 업데이트

## 모델 배분

| 역할 | 모델 | 사용 빈도 |
|------|------|----------|
| 오너 | Opus | L/D 티어, 최종 확인 시만 |
| 리뷰어 | Sonnet (서브에이전트) | 매 작업 완료 시 |
| 실행 - 구현 | Sonnet | 주력 |
| 실행 - 탐색/보고 | Haiku | 보조 |

## 스킬 구성안 (별도 레포 셋업 시)

| 스킬명 | 용도 |
|--------|------|
| `lm-dev` | 메인 워크플로우 (파이프라인 실행) |
| `lm-review` | 품질 리뷰 (체크리스트 대조) |
| `lm-rules` | TTS 규칙 관리/업데이트 |
| `lm-status` | 진행 상황 조회 |
| `lm-sample` | 샘플 생성 (과목/파일 지정) |
