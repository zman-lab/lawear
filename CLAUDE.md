## 핵심 규칙

### AI 자의적 해석 절대 금지 (최우선 규칙)
- 원본 PDF에 있는 정보만 사용
- "자연스럽게" 한다고 내용을 추가/변경 금지
- 법률 시험이라 한 글자가 달라도 틀린 답
- 의심스러우면 원본 표현 유지

### TTS 규칙
- 상세 규칙: `docs/handover/tts_rules.md`
- 기본 10개 + 사용자 피드백 11개 (R-01 ~ R-11)
- 규칙 충돌 시 R-09(자의적 해석 금지)가 최우선
- **게시판 #453 댓글 #462가 tts_rules.md보다 우선** (사용자 직접 작성)

### 출력 형식
- 문제 → 목차 → 답안 → diff → QA (고정)
- 목차는 원문 그대로 (요약/재구성 금지)
- diff 필수 제공 (사용자 최종 검수용)
- QA 필수 (듣는 사람 입장 검증)

## 팀 구조 — 이중 검증 체계

┌─────────────────────────────────────────────┐
│  학습 오너 (Opus) — 전략 + 최종 확인        │
│  - 과목별 전처리 전략 설계/판단             │
│  - 최종 품질 확인 (리뷰어가 놓친 것만 잡기) │
│  - "자연스러움" 등 판단이 필요한 부분 집중  │
│  - L/D 티어 의사결정                        │
│  - 학습 우선순위/스케줄 조율                │
│  - 전략 수정이 필요한 패턴 발견 시 규칙 업데이트 │
├─────────────────────────────────────────────┤
│  품질 리뷰어 (Sonnet 서브에이전트) — 1차 검증│
│  - 실행팀이 제출한 체크리스트 10개 자동 대조 │
│  - 전처리 규칙 적용 여부 기계적 검증        │
│  - 원본 PDF 대비 누락/왜곡 검출             │
│  - 체크리스트 미제출 → 즉시 반려            │
├─────────────────────────────────────────────┤
│  실행팀                                      │
│  - Sonnet: PDF 텍스트 추출, AI 전처리,      │
│    양식→줄글 재구성, TTS 텍스트 생성        │
│  - Haiku: 파일 탐색, 진행 상황 게시판 보고, │
│    단순 정규식 치환                          │
│  - ⚠️ 작업 완료 시 체크리스트 증빙 강제 제출│
└─────────────────────────────────────────────┘

### 이중 검증 흐름

실행팀(Sonnet) → 작업 완료 + 체크리스트 10개 증빙 제출
    ↓
리뷰어(Sonnet) → 1차 검증 (체크리스트 자동 대조, 기계적 확인) → 미통과 시 반려 + 사유
    ↓
오너(Opus) → 최종 확인 (판단 필요한 부분만 — 자연스러움, 맥락 적절성, AI 자의적 해석 여부)

### 모델 배분

| 작업 | 모델 | 이유 |
|------|------|------|
| 전처리 전략/설계 판단 | Opus | 규칙 충돌, 자의적 해석 경계 판단 |
| TTS 텍스트 변환 | Sonnet | 주력 실행 |
| 품질 리뷰 | Sonnet 서브에이전트 | 독립적 검증 |
| PDF 파일 탐색 | Haiku | 대량 파일 탐색 |
| 게시판 글/댓글 | Haiku | 비용 최소화 |

### 실행팀 필수 체크리스트 (10개)

작업 완료 시 아래 10개 증빙 제출. 하나라도 빠지면 반려.

1. 판례번호 제거 완료
2. 날짜 정규화 완료 (상대적 시간 표현 포함)
3. 꺽쇠 인용부호 제거 완료
4. 특수 띄어쓰기 정리 완료
5. 한자 병기 제거 완료 (인명 甲乙丙은 한글 변환)
6. 페이지 헤더/푸터 제거 완료
7. 빈칸/○○○ 처리 완료
8. 과목별 특화 전처리 적용 완료
9. 원본 대비 정보 누락 없음 확인
10. TTS로 읽었을 때 자연스러운 문장 확인
