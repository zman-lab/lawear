## 핵심 규칙

### PM은 디스패처 전용 — 직접 코드 수정 절대 금지! (반복 위반 주의)
- **le-dev 스킬 활성 시**: 메인 에이전트는 PM/디스패처 역할만
- **코드 읽기/수정/Edit/Write는 반드시 서브에이전트에 위임**
- **긴급해도 예외 없음** — 직접 하면 검증 누락으로 오히려 느려짐
- **dev-team Phase 1~6 워크플로우 준수**: 분석→설계→구현→QA→리뷰→dev-qa
- 이 규칙은 2번 위반 후 사용자가 명시적으로 추가 (2026-03-20)

### AI 자의적 해석 절대 금지 (최우선 규칙)
- 원본 PDF에 있는 정보만 사용
- "자연스럽게" 한다고 내용을 추가/변경 금지
- 법률 시험이라 한 글자가 달라도 틀린 답
- 의심스러우면 원본 표현 유지

### TTS 규칙
- 상세 규칙: `docs/handover/tts_rules.md`
- 기본 10개 + 사용자 피드백 11개 (R-01 ~ R-11)
- 규칙 충돌 시 R-09(자의적 해석 금지)가 최우선
- **게시판 #453 댓글 #462가 tts_rules.md보다 우선** (사용자 직접 작성)

### 출력 형식
- 문제 → 목차 → 답안 → diff → QA (고정)
- 목차는 원문 그대로 (요약/재구성 금지)
- diff 필수 제공 (사용자 최종 검수용)
- QA 필수 (듣는 사람 입장 검증)

## 팀 구조

메인 에이전트(PM/디스패처 전용) + 도메인팀 + `/dev-team` 공통 개발팀.

- **메인 에이전트**: 디스패처 전용 (코드 읽기/수정 안 함, 서브에이전트 디스패치)
- **도메인팀**: TTS 학습 전문가
  - 학습 오너(Opus) — 전처리 전략 설계/판단, 규칙 업데이트, L/D 의사결정
  - 품질 리뷰어(Sonnet) — 체크리스트 10개 자동 대조, 전처리 규칙 검증
  - TTS 실행(Sonnet) — PDF 텍스트 추출, AI 전처리, 양식→줄글 재구성, TTS 텍스트 생성
- **개발**: `/dev-team` 공통 개발팀 워크플로우 (Phase 1~6 + dev-qa 게이트)

### 이중 검증 흐름

1. TTS 실행(Sonnet) → 작업 완료 + 체크리스트 10개 증빙 제출
2. 품질 리뷰어(Sonnet) → 1차 검증 → 미통과 시 반려
3. 학습 오너(Opus) → 최종 확인 (자연스러움, 맥락, AI 자의적 해석 여부)

### 실행팀 필수 체크리스트 (10개)

작업 완료 시 아래 10개 증빙 제출. 하나라도 빠지면 반려.

1. 판례번호 제거 완료
2. 날짜 정규화 완료 (상대적 시간 표현 포함)
3. 꺽쇠 인용부호 제거 완료
4. 특수 띄어쓰기 정리 완료
5. 한자 병기 제거 완료 (인명 甲乙丙은 한글 변환)
6. 페이지 헤더/푸터 제거 완료
7. 빈칸/○○○ 처리 완료
8. 과목별 특화 전처리 적용 완료
9. 원본 대비 정보 누락 없음 확인
10. TTS로 읽었을 때 자연스러운 문장 확인
