## 핵심 규칙

### 메인 에이전트 역할 분리 (TTS 변환 vs 코드 수정)
- **TTS 변환(Lv.1/2/3)은 메인이 직접 수행** — 규칙/기준 게시글을 직접 읽은 컨텍스트로 변환
- **코드 수정만 서브에이전트 위임** — dev-team Phase 1~6 워크플로우 준수
- **TTS 변환 전 `/le-tts-load` 필수** — 규칙/기준 샘플 컨텍스트 로딩 (세션 1회)
- 이유: 서브에이전트는 PM이 프롬프트에 넘긴 것만 본다. 메인이 직접 규칙을 읽고 변환해야 품질이 보장됨 (2026-03-22)

### AI 자의적 해석 절대 금지 (최우선 규칙)
- 원본 PDF에 있는 정보만 사용
- "자연스럽게" 한다고 내용을 추가/변경 금지
- 법률 시험이라 한 글자가 달라도 틀린 답
- 의심스러우면 원본 표현 유지

### TTS 규칙
- 상세 규칙: `docs/handover/tts_rules.md`
- 기본 10개 + 사용자 피드백 규칙 (R-01 ~ R-23)
- 규칙 충돌 시 R-09(자의적 해석 금지)가 최우선
- R-23: 문제 1개씩 Lv.1/2/3 동시 변환 → 사용자 OK 후 다음 문제
- **TTS 변환 전 `/le-tts-load` 호출 필수** (규칙+기준 샘플 로딩)

### 출력 형식
- 문제 → 목차 → 답안 → diff → QA (고정)
- 목차는 원문 그대로 (요약/재구성 금지)
- diff 필수 제공 (사용자 최종 검수용)
- QA 필수 (듣는 사람 입장 검증)

## 팀 구조

메인 에이전트(TTS 변환 + PM) + `/dev-team` 공통 개발팀.

- **메인 에이전트**: TTS 변환은 직접, 코드 수정은 서브에이전트 위임
- **TTS 변환 흐름** (메인 에이전트 내 순차 실행):
  1. 작가 — PDF 원문 + 로드된 규칙/기준 샘플 기반으로 Lv.1/2/3 생성
  2. 체크리스트 10개 증빙 — 기본 전처리 기계적 검증
  3. R-01~R-23 적용 증빙 — 각 규칙별 적용 여부 + 해당 없으면 N/A
  4. 수험생 QA — 듣는 관점 자체 리뷰
  5. 판사 QA — 원본 대비 누락/R-09 위반 검증
  6. 사용자 검증 — 게시판에 Lv.1/2/3 + diff + 체크리스트 + R적용표 올려서 사용자 OK 대기
- **개발**: `/dev-team` 공통 개발팀 워크플로우 (Phase 1~6 + dev-qa 게이트)

### 실행팀 필수 체크리스트 (10개)

작업 완료 시 아래 10개 증빙 제출. 하나라도 빠지면 반려.

1. 판례번호 제거 완료

4. 특수 띄어쓰기 정리 완료
5. 한자 병기 제거 완료 (인명 甲乙丙은 한글 변환)
6. 페이지 헤더/푸터 제거 완료
7. 빈칸/○○○ 처리 완료
